# Mobile App Map — Kutumb Health (Expo + React Navigation)

> Generated by Genesis step-5. Stack: Expo 55, React Navigation 6, `@react-native-firebase/auth`, Axios.

---

## 1. Entry Points

### `index.ts`
Standard Expo entry: calls `registerRootComponent(App)` which internally calls `AppRegistry.registerComponent('main', () => App)`.

### `App.tsx`
Root component. Wraps everything in two providers then renders the navigator:
```
<AuthProvider>         ← global auth state (AuthContext)
  <StatusBar style="dark" />
  <RootNavigator />    ← decides which screen tree to show
</AuthProvider>
```

---

## 2. Navigation Structure

Three stacked navigator layers:

```
RootNavigator (NativeStack, headerShown: false)
├── "Auth"        → AuthNavigator          (when firebaseUser == null)
├── "Onboarding"  → OnboardingScreen       (when firebaseUser set but appUser.date_of_birth missing)
└── "App"         → AppNavigator           (when firebaseUser set and profile complete)
```

### AuthNavigator (`src/navigation/AuthNavigator.tsx`)
Native stack, no headers. Two screens:
- `Login` → `LoginScreen`
- `Register` → `RegisterScreen`

Navigation between them via `navigation.navigate('Register')` / `navigation.navigate('Login')`.

### AppNavigator (`src/navigation/AppNavigator.tsx`)
Bottom tab navigator with emoji icons. 5 tabs:

| Tab name  | Component       | Header title  | Emoji |
|-----------|-----------------|---------------|-------|
| `Home`    | `HomeScreen`    | Dashboard     | 🏠    |
| `Checkin` | `CheckinScreen` | Check-in      | ✏️    |
| `Medicines` | `MedicinesScreen` | Medicines  | 💊    |
| `AIChat`  | `AIChatScreen`  | Kutumb AI     | 🤖    |
| `Family`  | `FamilyScreen`  | Family        | 👨‍👩‍👧  |

Tab bar style: `COLORS.surface` background, 60px height, `COLORS.primary` active tint.

### RootNavigator routing logic
```ts
const isNewUser = firebaseUser && !appUser?.date_of_birth;

if (isLoading)      → spinner (COLORS.background, COLORS.primary indicator)
if (!firebaseUser)  → Auth stack
if (isNewUser)      → Onboarding screen
else                → App bottom tabs
```
Routing is **purely reactive** — no `navigation.navigate()` calls for auth transitions; changing `firebaseUser`/`appUser` in context triggers an automatic re-render of the correct stack.

## 3. Auth State Management (`src/store/AuthContext.tsx`)

### Library
`@react-native-firebase/auth` — the **native** Firebase SDK (not the JS web SDK). Token handling is built-in.

### State
```ts
firebaseUser: FirebaseAuthTypes.User | null  // raw Firebase auth object
appUser: User | null                         // backend user model (from /auth/login)
isLoading: boolean                           // true while onAuthStateChanged fires on boot
```

### Boot sequence
```
onAuthStateChanged fires
  → setFirebaseUser(fbUser)
  → if fbUser: fbUser.getIdToken() → login(token) → setAppUser(response.user)
  → setIsLoading(false)
```

### `signInWithEmail(email, password)`
```
auth().signInWithEmailAndPassword(email, password)
  → cred.user.getIdToken()
  → login(token)   // POST /auth/login
  → setAppUser
```

### `signUpWithEmail(email, password, name)`
```
auth().createUserWithEmailAndPassword(email, password)
  → cred.user.updateProfile({ displayName: name })
  → cred.user.getIdToken(true)   // forceRefresh=true
  → register(token)              // POST /auth/register
  → setAppUser({ ...response.user, name })   // patch name immediately
```

### `signOut()`
```
auth().signOut()
  → setAppUser(null)
  → setFirebaseUser(null)
```
After signOut, `onAuthStateChanged` fires again with null, so the two `set*` calls here are technically redundant but ensure immediate UI update.

### New-user detection
`appUser?.date_of_birth` being absent signals first-time user → Onboarding route. Once OnboardingScreen completes and the backend saves the date_of_birth, the next `login()` call (or manual refetch) will populate it and RootNavigator will switch to `App`.

### Local memory (AI memory stored on device)
Two exported helpers (used by AIChatScreen and OnboardingScreen):
```ts
loadLocalMemory(): Promise<Record<string, unknown>>
  → SecureStore.getItemAsync('kutumb_local_memory') → JSON.parse

saveLocalMemory(memory): Promise<void>
  → SecureStore.setItemAsync('kutumb_local_memory', JSON.stringify(memory))
```

`applyMemoryPatches(memory, patches)` — client-side JSON-patch-like engine. Supported ops: `update`, `add`, `remove`, `append_to_array`, `merge`. Traverses nested paths using `/`-separated keys (e.g., `"/user/conditions/0"`).

## 4. API Base URL

Configured in `src/constants/index.ts`:
```ts
export const API_BASE_URL = __DEV__
  ? 'http://10.0.2.2:8000/api/v1'          // Android emulator → host machine localhost
  : 'https://api.kutumb.health/api/v1';
```
`__DEV__` is a Metro/React Native global set to `true` in development builds, `false` in production.

## 5. API Client (`src/api/client.ts`)

Single Axios instance shared by all API modules:
```ts
const client = axios.create({
  baseURL: API_BASE_URL,
  timeout: 30_000,
  headers: { 'Content-Type': 'application/json' },
});
```

**Request interceptor** — injects Firebase ID token on every outgoing request:
```ts
const user = auth().currentUser;
if (user) {
  const token = await user.getIdToken();
  config.headers.Authorization = `Bearer ${token}`;
}
```
Token is fetched fresh every request (Firebase SDK caches and auto-refreshes internally, so this is cheap and always up-to-date).

**Response interceptor** — normalizes backend errors:
```ts
// On error: extract error.response?.data?.detail ?? error.message ?? 'Something went wrong'
// Re-throw as new Error(message)
```
This means every `catch` block in screen code receives a plain `Error` with the backend's `detail` field as `.message`.

## 6. API Modules

## 9. Screen Details

### `LoginScreen` (`src/screens/auth/LoginScreen.tsx`)
- **Purpose:** Email + password sign-in
- **Inputs:** email (email keyboard, no caps), password (secureTextEntry)
- **Action:** calls `signInWithEmail(email, password)` from AuthContext
- **Navigation:** entirely handled by RootNavigator reacting to auth state change — no explicit `navigate()` call
- **Error handling:** `Alert.alert('Login failed', err.message)`
- **Link:** "New here? Create an account" → `navigation.navigate('Register')`

### `RegisterScreen` (`src/screens/auth/RegisterScreen.tsx`)
- **Purpose:** Create new account (name, email, password)
- **Validation:** All 3 fields required; password min 6 chars
- **Action:** calls `signUpWithEmail(email, password, name)` from AuthContext
- **Navigation:** automatic via auth state; new user has no `date_of_birth` → Onboarding
- **Link:** "Already have an account? Sign in" → `navigation.navigate('Login')`

### `OnboardingScreen` (`src/screens/auth/OnboardingScreen.tsx`)
- **Purpose:** AI-guided first-time setup — collects user health profile through conversation
- **Initial message:** Namaste greeting from Kutumb AI, asks for name
- **Stage machine:** starts at `'welcome'`, advances via `res.next_stage` from backend
- **API call:** `startOnboarding(stage, userText, conversationHistory)` → `/ai/session/onboard`
- **Memory updates:** `applyMemoryPatches(localMemory, res.patches)` then `saveLocalMemory()` on every turn
- **UI:** FlatList of chat bubbles (user=right/primary, assistant=left/surface border), typing indicator, multiline input with send button
- **Completion:** When onboarding fills `date_of_birth` on the backend profile, the next `login()` call populates `appUser.date_of_birth` and RootNavigator switches to App tabs

### `HomeScreen` (`src/screens/home/HomeScreen.tsx`)
- **Purpose:** Main dashboard — health snapshot
- **Data loaded:** `Promise.allSettled([getHealthScores(), getAdvisories(), getTodaySchedule()])`  
  Fails silently if any request fails (API may be down in dev)
- **Sections:**
  1. **Greeting** — time-based (morning/afternoon/evening) + first name; pill badge for pending/overdue doses
  2. **Overall health card** — green card with large `overall` score + 4 organ `ScoreRing` components (circular borders in organ-specific colors)
  3. **Empty state** — if no scores, prompt to do first check-in
  4. **Today's medicines** — adherence progress bar + first 4 DoseSchedule items (overdue rows highlighted red)
  5. **Advisories** — colored cards (info=blue, warning=yellow, critical=red) with left border accent
- **Pull to refresh:** re-runs `load()`

### `CheckinScreen` (`src/screens/checkins/CheckinScreen.tsx`)
- **Purpose:** Daily health check-in form (create or update today's entry)
- **On mount:** calls `getTodaysCheckin()` — if data exists, pre-fills all fields and sets `existing=true`
- **EmojiSlider component:** row of tap targets (1–5 or custom range) using emoji labels
- **Fields:**
  - Mood: 😢😕😐🙂😄 (required — blocks submit)
  - Energy: 😴🥱😐⚡🚀
  - Pain: 0️⃣1️⃣2️⃣3️⃣4️⃣
  - Stress: 😌🙂😐😟😰
  - Sleep hours: 4–10h picker
  - Sleep quality: emoji slider
  - Hydration: 1–8 glasses picker
  - Symptoms: 10 chip options (multi-select toggle)
  - Notes: free-text multiline
- **Submit:** `createCheckin(payload)` → shows success Alert
- **Button label:** "Save check-in" or "Update check-in" based on `existing`

### `MedicinesScreen` (`src/screens/medicines/MedicinesScreen.tsx`)
- **Purpose:** Medication management — schedule view and medicine library
- **Tabs:** "Today's schedule" / "All medicines" (local state toggle, same underlying data)
- **Data loaded:** `Promise.allSettled([getTodaySchedule(), listMedicines()])`
- **Today tab:**
  - Adherence progress bar at top
  - FlatList of `DoseCard` — shows status dot (color-coded), medicine name, dosage+time
  - Pending/overdue cards show "Taken" and "Skip" action buttons
  - "Taken" calls `logDose(medicineId, 'taken', scheduledTime)` then re-loads
  - Overdue cards get red border/background
- **All medicines tab:**
  - FlatList of `MedicineCard` — shows 🚑 for emergency meds, 💊 otherwise
  - Refill alert row (warning color) if `refill_alert=true`
  - Stock count display

### `AIChatScreen` (`src/screens/ai/AIChatScreen.tsx`)
- **Purpose:** Conversational AI health assistant
- **On mount:** `loadLocalMemory()` from SecureStore to populate `localMemory`
- **Message sending flow:**
  1. Build `newHistory = [...conversationHistory, userMsg]`
  2. `sendMessage({ message, conversation_history: newHistory, memory_file: localMemory })`
  3. Append assistant reply to messages list
  4. If `fired_triggers` returned → display as colored `TriggerAlert` banners above chat
  5. If `patches` returned → `applyMemoryPatches` + `saveLocalMemory`
- **TriggerAlert:** border-colored cards (critical=red, warning=orange, info=green), shown above FlatList
- **Suggested prompts:** shown as chips when only the welcome message exists (1 message in list). Tapping a chip sets the input text.
- **Welcome message:** "I have your complete health profile and memory. Ask me anything…"
- **Memory loop:** every turn reads from local memory + sends it to backend + applies patches back — creates persistent AI memory across sessions without server-side session storage

### `FamilyScreen` (`src/screens/family/FamilyScreen.tsx`)
- **Purpose:** View and manage family member connections
- **Data:** `listFamilyMembers()` → `{ members: FamilyMember[] }`
- **Info bar:** shows count + linked count
- **MemberCard:** emoji avatar (relation-based lookup), name, relation, linked/unlinked badge, permission badge
- **Invite button:** shown only for unlinked members; calls `inviteFamilyMember(memberId)` if phone exists, else shows alert to add phone first
- **Empty state:** message suggests using AI chat to add members: "Add my father Suresh, age 65"
- **Pull to refresh:** standard pattern

## 10. Auth + API Flow Summary

```
User opens app
  → AuthProvider mounts
  → onAuthStateChanged fires
  → [if returning user] fbUser present → getIdToken() → POST /auth/login → appUser set
  → isLoading false
  → RootNavigator renders correct tree

User logs in (LoginScreen)
  → Firebase signInWithEmailAndPassword
  → getIdToken()
  → POST /auth/login → appUser set
  → RootNavigator auto-switches to App or Onboarding

Any API call (e.g. getHealthScores)
  → client.get('/health/me/scores')
  → request interceptor: auth().currentUser.getIdToken()
  → injects: Authorization: Bearer <firebase-id-token>
  → backend verifies with Firebase Admin SDK
  → response returned

AI chat turn
  → AIChatScreen sends { message, conversation_history, memory_file }
  → POST /ai/session/message
  → backend: assembles context (profile + memory + history + health data)
  → Claude generates reply + patches + triggers
  → client applies patches to SecureStore
  → fired triggers rendered as banners
```

## 11. Key Architectural Observations

1. **Stateless AI sessions on mobile**: No session ID is stored. Every turn sends the full `conversation_history` array + the local `memory_file`. Memory patches are persisted in SecureStore (device-local), not the backend.

2. **Token injection is transparent**: All screens use the API modules without thinking about auth — the Axios interceptor silently handles token attachment via `@react-native-firebase/auth`.

3. **Two distinct user objects**: `firebaseUser` (Firebase SDK object, never serialized) and `appUser` (backend User model). The new-user gate (`date_of_birth`) is on `appUser`, not Firebase custom claims.

4. **Dev URL is Android-emulator-aware**: `10.0.2.2` maps to `localhost` on Android emulator. iOS simulator uses `localhost` directly — but the current constant only handles Android. iOS dev would need `localhost`.

5. **Error normalization**: The Axios response interceptor guarantees all errors are `Error` instances with `.message` = backend `detail` string. Screen code uniformly uses `err instanceof Error ? err.message : 'fallback'`.

6. **Pull-to-refresh pattern**: HomeScreen, MedicinesScreen, and FamilyScreen all use `useCallback`-wrapped `load()` + `RefreshControl` — identical pattern throughout.

7. **No navigation param passing**: No screen receives data via route params. All data fetching is self-contained within each screen on mount/focus. This simplifies navigation but means duplicate API calls when switching tabs.

8. **Medicine addition via AI**: FamilyScreen and MedicinesScreen empty states both hint users to use AI chat to add data ("Add my father Suresh, age 65" / "add medicines via AI chat"). The AI session triggers on the backend handle writing to Firestore.
